# 6 種任務類型模板

> 來源：169 個 git commits 歷史分析，提取高效溝通結構

## A. 新增功能/模組

```
任務：[一句話描述功能]

背景：
- 這個功能要解決什麼問題
- 現有哪些相關模組可以擴展（先搜尋，避免重複建立）

環境：[local / VPS / cron]

影響範圍：
- [ ] 需要修改的現有檔案
- [ ] 需要新建的檔案（確認沒有現成可擴展的）
- [ ] 下游依賴（誰會用到這個功能）

驗收條件：
- [ ] 具體的測試方式（執行什麼指令、預期什麼輸出）
- [ ] 邊界案例

參考：[指向現有類似實作]
```

**真實教訓 — FRED 整合（7 commits → 本可 1）：**
只說「整合 FRED API」，沒有列出 pipeline 每層影響，導致收集→轉發→處理→驗證逐層補漏。

## B. Bug 修復

```
Bug 描述：
- 症狀：[具體錯誤訊息或異常行為]
- 發生時機：[什麼情況下觸發]
- 頻率：[每次 / 偶爾 / 特定條件]

環境：
- 發生在哪裡：[local / VPS / cron]
- Node 版本 / OS / 相關版本資訊

重現步驟：
1. [步驟 1]
2. [步驟 2]
3. [預期行為 vs 實際行為]

已嘗試：[已經試過什麼，排除了什麼可能]

錯誤日誌：
[貼上完整的 error output，不要截斷]

相關檔案：[你認為可能相關的檔案路徑]

驗收條件：
- [ ] Bug 修復後的預期行為
- [ ] 怎麼驗證修復成功（指令 + 預期輸出）
```

**真實教訓 — industryThemes debug 迴圈（5 commits → 本可 1）：**
只說「產業熱點不見了」，沒有提供資料流追蹤，導致 3 輪盲目 debug。

## C. 整合新資料源/API

```
任務：整合 [API 名稱] 到 [目標模組]

API 資訊：
- 端點：[URL]
- 認證：[API Key 變數名]
- 已知限制：[速率限制、延遲、不支援的功能]
- 回應格式：[JSON 欄位範例]

Pipeline 影響（逐層檢查）：
- [ ] 收集層（新建 collector 或擴展現有）
- [ ] 轉發層（phase 間的資料傳遞）
- [ ] 處理層（validator、scorer 等）
- [ ] 配置層（config.json 閾值）
- [ ] 渲染層（日報顯示格式）
- [ ] 推播層（推送管道）

環境：[local / VPS / cron]
超時設定：[建議值]

驗收條件：
- [ ] 每層的預期輸出
- [ ] 錯誤降級行為（API 不可用時怎麼辦）
```

## D. 部署/環境操作

```
任務：[部署/同步/環境修改]

目標環境：
- 在哪裡操作：[local → VPS / VPS 內部 / cron 排程]
- VPS 資源限制：1.1GB RAM、2 core CPU

操作內容：
- [ ] 要同步的檔案清單
- [ ] 要執行的指令序列

風險評估：
- 是否影響正在運行的服務？
- 是否需要重啟？
- 是否可回滾？

SSH 注意事項：
- 指令必須逐一發送，不可並行
- 避開 cron 高峰期（每 5 分鐘整點）
- 大量文字用 base64 編碼

驗收條件：
- [ ] 部署後的健康檢查指令
- [ ] 預期的服務狀態
```

## E. 配置/閾值調整

```
任務：調整 [配置項目]

目前值：[現在的設定]
目標值：[想改成什麼]
原因：[為什麼要改，基於什麼數據]

影響範圍：
- [ ] 哪些模組會讀取這個配置
- [ ] 是否影響其他環境（local vs VPS）

參考數據：
- 實際觀測值：[提供真實數據]
- 極端情況：[最大值/最小值]

驗收條件：
- [ ] 修改後的測試方式
- [ ] 預期的正常範圍
```

**真實教訓 — system-load 閾值（4 commits → 本可 1）：**
沒有提供 VPS 實際 load 數據，猜測閾值導致反覆調整。

## F. 重構/跨模組修改

```
任務：[重構目標]

範圍：
- [ ] 受影響的檔案清單（全部列出）
- [ ] 是否改變對外介面（exports、API）
- [ ] 是否需要更新 import/require

策略：
- 是否分階段進行？（建議大重構分 phase）
- 每個 phase 的範圍和驗收點

向後相容：
- 是否有其他模組依賴被改的介面？
- 是否需要暫時保留舊介面？

驗收條件：
- [ ] 每個 phase 的測試方式
- [ ] 整體完成後的 E2E 驗證
```

## 驗收條件寫法建議

好的驗收條件應包含：

1. **在哪裡** — 指定檔案或章節
2. **做什麼** — 明確動作
3. **怎麼確認** — 指令或輸出

**好的例子：**

- [ ] 在 `config.json` 新增 `"timeout": 30`
      驗收：`node -e "console.log(require('./config.json').timeout)"` 輸出 30

- [ ] 在 `collector.js` 移除硬編碼 URL
      驗收：`grep API_URL collector.js` 有輸出

- [ ] 在 `README.md` 補上 rollback 說明
      驗收：`grep rollback README.md` 有輸出

**不夠好的例子：**

- [ ] 測試通過
- [ ] 功能正常
- [ ] 優化效能
